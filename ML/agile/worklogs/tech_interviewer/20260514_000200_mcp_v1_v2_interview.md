## Level 1 — Basic Recall

### Q1. What does MCP stand for? In one sentence, what problem does it solve?

<details>
<summary>Hint (read after you answer)</summary>
Model Context Protocol. It solves the problem of tightly coupling AI agents to specific tool implementations — instead of hard-coding how tools work, MCP gives a standard way for agents to discover and call tools dynamically at runtime.
</details>

---

### Q2. In our v1 implementation, we had two Python files in one FastAPI app. What was the responsibility of each file?

<details>
<summary>Hint</summary>
- `mcp_server.py` — MCP server. Exposes the `add_two_numbers` tool over SSE (Server-Sent Events). Handles `/mcp/sse` and `/mcp/messages` endpoints.
- `calculation_api.py` — Agent endpoint. Receives `{a, b}`, asks Azure OpenAI which tool to call, executes the tool, sends the result back to OpenAI, returns the final answer.
</details>

---

### Q3. What is SSE? Why does MCP use it instead of a regular REST POST?

<details>
<summary>Hint</summary>
SSE = Server-Sent Events. It is a persistent, one-way HTTP connection where the server can push multiple events to the client over time. MCP uses it because tool discovery and tool calls are an ongoing protocol exchange — not a single request/response. The client connects once via SSE and then sends messages, receiving streamed responses.
</details>

## Level 2 — Conceptual Understanding

### Q4. The architect said v1 is "not a true MCP implementation." What exactly was wrong?

<details>
<summary>Hint</summary>
In `calculation_api.py`, the line:
```python
from mcp_server import add_two_numbers
```
This is a direct Python import. The agent calls the function the same way you call any function in another module — there is no protocol involved. The MCP server's SSE endpoint (`/mcp/sse`) was NEVER called by the agent at runtime. MCP was only used on the server side (exposing the tool) but not on the client side (consuming the tool).
</details>

---

### Q5. In v1, Azure OpenAI was involved in tool calling. But the architect still said v1 is not MCP. Why? What is the difference between OpenAI function calling and MCP?

<details>
<summary>Hint</summary>
OpenAI function calling = the model decides WHICH tool to call and formats the arguments. That is the AI's job.
MCP = the protocol that delivers the tool call to the tool server and returns the result. That is the transport layer's job.

In v1, OpenAI correctly decided to call `add_two_numbers(a=3, b=5)`. But the delivery of that call was done by Python import — not MCP. True MCP would use `mcp.ClientSession` to connect to the SSE endpoint and send the call via the MCP protocol.

You can use OpenAI function calling AND MCP together — they solve different problems.
</details>

---

### Q6. The `add_two_numbers` tool definition was hardcoded in `calculation_api.py` in v1. Why is that a problem? What should v2 do instead?

<details>
<summary>Hint</summary>
Hardcoded tool definitions mean: if the MCP server adds a new tool, the agent code must also be manually updated. The two are tightly coupled.

In v2, the agent should call `session.list_tools()` at runtime to fetch the available tools from the MCP server dynamically. This means:
- New tools appear automatically without changing the agent
- Agent and MCP server can be on different machines / services
- True decoupling
</details>

## Level 3 — Edge Cases & Architecture

### Q7. In v1, the MCP server and agent are in the same FastAPI app (same process). What would break if you moved the MCP server to a separate machine?

<details>
<summary>Hint</summary>
In v1: `from mcp_server import add_two_numbers` — this Python import requires both files to be in the same process. If you move the MCP server to a different machine, this import fails entirely. The whole v1 architecture collapses.

In v2 with `mcp.ClientSession`: the agent connects to an HTTP URL (e.g., `http://mcp-server-host:8001/mcp/sse`). The server can be anywhere — different process, different machine, different cloud region. That is the entire point of MCP.
</details>

---

### Q8. Azure OpenAI requires a `deployment` name in the `model=` field, not the model name. Why? How is this different from regular OpenAI?

<details>
<summary>Hint</summary>
Regular OpenAI: you call a shared service at `api.openai.com`. The model name (`gpt-4o-mini`) is enough to route your request.

Azure OpenAI: you deploy a specific model to your own Azure resource under a custom deployment name. The deployment name is how Azure routes the request — the model name alone is ambiguous because you could deploy multiple versions. In our case:
- Deployment name: `etna-hr05-openai-cus001-hr05_gpt_4o_mini`
- Actual model: `gpt-4o-mini`

You pass the deployment name in `model=`, not `gpt-4o-mini`.
</details>

---

### Q9. What will v2's `calculation_api.py` do differently — step by step?

<details>
<summary>Hint</summary>
v2 flow:
1. Open an SSE connection to `http://localhost:8000/mcp/sse` using `mcp.ClientSession`
2. Call `session.list_tools()` — fetch available tools dynamically from MCP server
3. Convert MCP tool definitions to OpenAI tool format
4. Send request to Azure OpenAI with those tools
5. Azure OpenAI decides to call `add_two_numbers`
6. Agent calls `session.call_tool("add_two_numbers", {"a": 3, "b": 5})` via MCP protocol — NOT a Python import
7. MCP server executes the function, returns result via SSE
8. Agent sends result back to Azure OpenAI for final answer

Key difference: steps 2, 6, and 7 are via MCP protocol — not Python imports.
</details>
